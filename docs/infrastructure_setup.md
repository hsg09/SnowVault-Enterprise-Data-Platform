# Multi-Cloud Infrastructure Setup Guide

## Executive Summary

Before the internal pipelines and Terraform automation (IaC) can run for the **SnowVault Enterprise Data Platform**, a series of foundational, external cloud environments must be bootstrapped. This guide provides the step-by-step instructions for establishing the pre-requisite infrastructure across Snowflake, AWS, Azure, GCP, Kafka, and GitHub.

---

## 1. Snowflake Initial Bootstrap

To enable Terraform to manage your Snowflake ecosystem, an initial Service Account with localized `SYSADMIN` and `SECURITYADMIN` privileges must be created. This explicitly bypasses password-auth in favor of mandatory RSA-keypair authentication.

**Prerequisite:** Generate a 2048-bit RSA Private/Public Key via OpenSSL.
```bash
openssl genrsa 2048 | openssl pkcs8 -topk8 -inform PEM -out rsa_key.p8 -nocrypt
openssl rsa -in rsa_key.p8 -pubout -out rsa_key.pub
```

**Snowflake Execution (Run as `ACCOUNTADMIN`):**
```sql
USE ROLE ACCOUNTADMIN;

-- Create the Terraform Deployer Role
CREATE ROLE TERRAFORM_DEPLOYER;
GRANT ROLE SYSADMIN TO ROLE TERRAFORM_DEPLOYER;
GRANT ROLE SECURITYADMIN TO ROLE TERRAFORM_DEPLOYER;

-- Create the Service Account
CREATE USER TERRAFORM_SVC_USER
    DEFAULT_ROLE = TERRAFORM_DEPLOYER
    RSA_PUBLIC_KEY = 'MIIBIjANBgkqh... (paste your rsa_key.pub contents without borders)';

GRANT ROLE TERRAFORM_DEPLOYER TO USER TERRAFORM_SVC_USER;
```

## 2. Multi-Cloud Storage Setup

The platform ingests massive datasets natively from all three major cloud providers. Storage must be provisioned and secured prior to Terraform Integration deployments.

### 2.1 Amazon Web Services (S3)
1. **Create S3 Storage:** Provision `s3://your-company-data-lake-raw/` in the `us-east-1` region.
2. **Create IAM Role:** Create an AWS IAM Role (`SnowflakeS3IntegrationRole`) granting `s3:GetObject` and `s3:PutObject` over the raw bucket.
3. **Establish Trust:** Once Terraform creates the `S3_RAW_INTEGRATION` inside Snowflake, copy the `STORAGE_AWS_IAM_USER_ARN` generated by Snowflake into your AWS IAM Role's **Trust Relationship Policy** to securely link the two environments.

### 2.2 Microsoft Azure (ADLS Gen 2)
1. **Create Storage Account:** Create an ADLS Gen 2 compliant storage account `yourstorageaccount` and a container named `raw`.
2. **Provision Managed Identity:** Assign the "Storage Blob Data Contributor" role to the Azure Tenant. 
3. **App Approval:** Execute the Admin Consent URL inside Azure Active Directory using the `AZURE_CONSENT_URL` output generated after Terraform applies the `ADLS_RAW_INTEGRATION`.

### 2.3 Google Cloud Platform (GCS)
1. **Provision Bucket:** Create a GCS bucket named `gs://your-company-gcs-raw/`.
2. **Assign Role:** Once the `GCS_RAW_INTEGRATION` is mapped via Terraform, note the `STORAGE_GCP_SERVICE_ACCOUNT` deployed by Snowflake.
3. **Grant IAM:** Go into the GCP Console and grant the Snowflake-provided service account the `Storage Object Admin` role over the bucket.

## 3. Streaming Ingestion (Kafka / Confluent)

To support low-latency Data Vault landing capabilities, the Snowpipe Streaming connector natively executes into the `RAW_VAULT` ecosystem.

1. **Connector Installation:** Install the `snowflake-kafka-connector` to your Confluent Cloud or self-hosted Kafka Connect cluster.
2. **Role Requirements:** The Kafka Connector utilizes the `LOADER` platform role generated by Terraform. 
3. **Configuration:** Inject the following mandatory config mapping into your Kafka JSON:
```json
{
  "connector.class": "com.snowflake.kafka.connector.SnowflakeSinkConnector",
  "snowflake.topic2table.map": "ecommerce_events:RAW_EVENTS",
  "snowflake.private.key": "<Inject Secondary RSA Key>",
  "snowflake.database.name": "RAW_VAULT",
  "snowflake.schema.name": "ECOMMERCE",
  "snowflake.role.name": "LOADER",
  "snowflake.ingestion.method": "SNOWPIPE_STREAMING"
}
```

> [!CAUTION]
> Ensure the Kafka Service Account uses a completely isolated secondary RSA-key pair specifically tied to the `LOADER` role. Do not reuse the `TERRAFORM_SVC_USER` key for data streaming.

## 4. GitHub Actions CI/CD Environment

For the workflows defined in `.github/workflows/` to function, the repository requires strict Secret initialization.

Navigate to **Settings > Secrets and variables > Actions** in the GitHub UI and configure the following absolute requirements:

| Secret Name | Value Description |
|---|---|
| `SNOWFLAKE_ACCOUNT` | Your Snowflake account locator (e.g., `xy12345.us-east-1`) |
| `SNOWFLAKE_USER` | The core service user (`TERRAFORM_SVC_USER`) |
| `SNOWFLAKE_PRIVATE_KEY` | The raw contents of `rsa_key.p8` (including header/footer boundaries) |
| `SNOWFLAKE_PRIVATE_KEY_PASSPHRASE` | Only necessary if you encrypted your `.p8` key during creation. Leave empty otherwise. |

> [!TIP]
> After establishing these 4 variables, you can immediately push to the `main` branch. GitHub Actions will dynamically authenticate via the RSA key, authorize Terraform to build all Multi-Cloud architectures, and authorize dbt to parse and execute all underlying database objects.
